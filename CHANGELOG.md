## [1.4.1] - 2026-06-10

### Исправлено (по живому тесту голосовых)
- **Модель открещивалась от голосовых.** Транскрипция работала, но Claude не знал, что
  `[голосовое сообщение]` — это расшифровка, и отвечал «не вижу голосовое». Добавлена
  подсказка в системный промпт (при `VOICE_ENABLED` + ключе).
- **Веб-поиск игнорировался.** На «Запроси стоимость нефти» Haiku гадал по памяти, не вызывая
  web_search. Hint переписан директивно: для цен/курсов/новостей — обязательный вызов.
- **Шаблон снова прорывался** («Краткий вывод/Критика/Уверенность» в 02:56). Имитация десятков
  старых ответов в истории пересиливала запрет. В `STYLE_ANCHOR` добавлена причина не имитировать:
  шаблонные ответы в истории — старый исправленный дефект, не образец. Та же прививка — промптам
  саммаризации (свежая сводка начиналась с «Итог:»). Отравленные сводки диалога удалены из
  `Summaries` (производные данные, перегенерятся).

## [1.4.0] - 2026-06-10

### Добавлено
- **Распознавание голосовых сообщений.** Claude API аудио не принимает, поэтому голосовые
  транскрибируются через OpenAI STT (`gpt-4o-mini-transcribe`, ~$0.003/мин) одним
  HTTPS-вызовом через `requests` — новых зависимостей и слоёв нет.
  - `telegram_utils.get_file_bytes()` — универсальное скачивание файла Telegram (до 20 МБ).
  - `worker_lambda._transcribe_voice()` — скачивание + multipart-запрос к
    `/v1/audio/transcriptions`; `.oga` маппится в `ogg`.
  - Транскрипт подставляется в `text` с маркером `[голосовое сообщение]` ДО сохранения
    в историю и mention-детекции — попадает и в контекст, и в проверку режима.
  - В личке при неудачной транскрипции бот честно просит продублировать текстом
    (в группах — тихо пропускает).
  - Env: `VOICE_ENABLED` (deflt 1), `OPENAI_API_KEY` (обязателен для фичи),
    `STT_MODEL` (deflt `gpt-4o-mini-transcribe`), `VOICE_MAX_DURATION_SEC` (deflt 300).

## [1.3.2] - 2026-06-09

### Изменено (токены / память — оптимизация стоимости)
- **Троттлинг краткой сводки.** `summarize_history()` в STEP8 регенерилась **на каждом**
  сообщении (лишний вызов Claude каждый ход). Теперь — не чаще раза в `SUMMARY_MIN_INTERVAL_SEC`
  (по умолч. 600 с) через `dynamo_utils.get_latest_summary_item()` (timestamp). Убирает ~85%
  вторых вызовов модели.
- **Дата/время вынесены в конец системного промпта.** Раньше стояли в начале — это волатильный
  префикс, который сломал бы prompt caching. Теперь `BASE_SYSTEM_PROMPT` идёт первым (стабильный
  кэшируемый префикс), дата/время — в самом конце.
- **`MAX_CONTEXT_TOKENS`: 20000 → 10000** (env). Меньше реплея истории каждый ход → дешевле вход
  и слабее имитация формата; старший контекст добирается сводкой.

### Исправлено
- **Мёртвый долгосрочный профиль.** Ветка `len(full) >= 50` не срабатывала никогда (`full`
  бралась с `limit=24`), поэтому `create_long_term_summary`/`extract_topics` не вызывались —
  отсюда пустые `long_term_summary`/`last_topics`. Теперь профиль обновляется раз в
  `LONG_TERM_EVERY` (50) сообщений по `profile.message_count`, только в private, на отдельной
  выборке истории.

## [1.3.1] - 2026-06-09

### Исправлено
- **Петля шаблонного стиля.** Бот скатывался в жёсткий формат ответов
  («Краткий вывод / Критика / Альтернативы / Итог / Уверенность», начало с «Ясно —»):
  модель имитировала собственный прошлый транскрипт, а саммари усиливали эффект
  («Итог: …»). Принцип фикса — **«память хранит факты, а стиль задаёт только промпт»**:
  - `worker_lambda`: добавлен **стиль-файрвол** `STYLE_ANCHOR` — последний блок системного
    промпта, явно запрещающий копировать формат истории и перечисляющий анти-паттерны.
  - `claude_utils`: промпты `summarize_history` и `create_long_term_summary` переписаны на
    **нейтральную фактическую прозу** (без заголовков/секций/«Итог») — память больше не несёт
    шаблон.
  - `BASE_SYSTEM_PROMPT` и объём истории/контекста не менялись (обсуждается отдельно).

## [1.3.0] - 2026-06-09

Ветка **v8**. Развивает механизм Tool Use из 1.2.0 четырьмя новыми возможностями.
Деплой затрагивает только `worker_lambda` (код); слои и `webhook_lambda` не меняются.

### Добавлено

#### Веб-поиск (web_search)
- Подключён серверный инструмент Anthropic `web_search_20250305` — Claude сам ищет
  актуальные факты в интернете. Выполняется на стороне API (без своего бэкенда).
- Управление через env: `WEB_SEARCH_ENABLED` (по умолчанию `1`), `WEB_SEARCH_MAX_USES` (3).
- Обрабатывается `stop_reason == "pause_turn"` для долгих поисков.

#### Долговременная память через инструменты (remember/forget)
- Новые клиентские инструменты `remember_fact` и `forget_fact`: модель осознанно
  сохраняет/удаляет устойчивые факты о собеседнике (не дожидаясь авто-саммаризации).
- Хранение в `Users.profile.facts` (новое поле). Функции `dynamo_utils`:
  `get_user_facts()`, `add_user_fact()` (с дедупликацией), `remove_user_facts()`
  (по подстроке или очистка всего).
- Факты подмешиваются в системный контекст: блок «Что важно помнить» (private)
  и «Важно помнить о текущем авторе» (группы).

#### Распознавание изображений (зрение)
- `_parse_update()` извлекает фото (`photo`, крупнейший размер в пределах ~3.5 МБ)
  и картинки-документы (`document` с `image/*`).
- `telegram_utils.get_file_base64()` скачивает файл Telegram → base64.
- Изображение подмешивается image-блоком в последнее сообщение пользователя.
- Управление через env: `VISION_ENABLED` (по умолчанию `1`).

#### Прогноз погоды
- Инструмент `get_forecast(city, day=today|tomorrow)` поверх OpenWeatherMap
  `/forecast` (5 дней / 3 часа): сводка «от … до …°C, основное явление».

### Изменено
- `claude_utils._ensure_alternation()` и `num_tokens_from_messages()` корректно
  работают с мультимодальным контентом (список блоков text/image).
- `_chat()`: graceful-degradation — при ошибке с инструментами повтор без серверных,
  затем без всех инструментов; защита от зацикливания (≤8 итераций tool use).
- Роутер инструментов вынесен в замыкание `_make_tool_executor(user_id)` —
  инструменты памяти знают текущего пользователя.
- `requirements.txt`: `anthropic>=0.49.0` (web search + image blocks).

## [1.2.0] - 2026-02-13

### Добавлено

#### Дата/время в контексте модели
- Текущая дата и время МСК автоматически передаётся в каждый запрос к Claude
- Модель корректно отвечает на вопросы "какое сегодня число", "на завтра" и т.д.

#### Погода через Claude Tool Use
- Реализован механизм **Tool Use** в `claude_utils.py`:
  - `_chat()` поддерживает цикл вызова инструментов (`stop_reason == "tool_use"`)
  - `generate_response()` принимает параметры `tools` и `tool_executor`
- Инструмент `get_weather` через OpenWeatherMap API:
  - Claude сам извлекает название города из сообщения пользователя
  - Если город не указан — используется `WEATHER_DEFAULT_CITY` (по умолчанию Moscow)
  - Возвращает температуру, ощущаемую температуру, влажность, скорость ветра
- Новые переменные окружения: `OPENWEATHERMAP_API_KEY`, `WEATHER_DEFAULT_CITY`

#### Обработка отредактированных сообщений
- `webhook_lambda.py` и `worker_lambda.py` теперь обрабатывают `edited_message`
- Бот реагирует на упоминание в отредактированном сообщении

#### Деплой через AWS CLI
- Установлен AWS CLI v2 и настроен IAM пользователь для деплоя
- Добавлена переменная `MESSAGES_TABLE` (ранее отсутствовала в env vars)
- Корректный `BASE_SYSTEM_PROMPT` через Python/boto3 (решена проблема кодировки)

### Исправлено
- **SQS InvalidParameterValue**: атрибуты с пустым значением (`thread_id`, `user_id` для каналов) теперь не отправляются в SQS
- **DynamoDB ValidationException**: при обновлении профиля старых пользователей (без поля `profile`) теперь автоматически инициализируется структура профиля и повторяется запрос — затрагивает `update_user_profile()` и `update_user_names()`

## [1.1.0] - 2025-01-20

### Добавлено

#### Персонализация и долгосрочная память
- **Извлечение полных данных пользователя** из Telegram API (`first_name`, `last_name`)
- **Структура User Profile** в DynamoDB:
  - `first_name`, `last_name` - реальные имена пользователей
  - `communication_style` - автоматически определяемый стиль общения
  - `interests` - список интересов (автоматическое извлечение)
  - `long_term_summary` - долгосрочная сводка о пользователе
  - `last_topics` - последние обсуждаемые темы
  - `message_count` - счётчик сообщений

#### Новые функции в dynamo_utils.py
- `update_user_names()` - обновление имён при изменении в Telegram
- `update_user_profile()` - обновление долгосрочной информации
- `get_user_profile()` - получение профиля пользователя
- Расширена `save_user()` для инициализации profile структуры

#### Улучшенная саммаризация (claude_utils.py)
- `summarize_history()` теперь:
  - Использует 30 сообщений вместо 16 (+87%)
  - Выделяет 600 токенов вместо 200 (+200%)
  - Сохраняет имена и стиль общения участников
  - Принимает опциональный `user_context`
- `create_long_term_summary()` - создание профиля пользователя:
  - Анализирует последние 60 сообщений
  - Определяет стиль общения, интересы, проекты
  - Генерирует 1-2 абзаца профиля (400 токенов)
- `extract_topics()` - извлечение ключевых тем:
  - Анализирует последние 10 сообщений
  - Возвращает до 5 ключевых тем/слов

#### Обогащённый контекст для модели (worker_lambda.py)

- Персональный блок в system prompt:

**Для групповых чатов:**

#### Автоматическое обновление профилей

- Долгосрочная сводка создаётся после 50+ сообщений

## [1.0.0] - 2025-01-15

### Добавлено

#### Базовая функциональность
- Асинхронная архитектура через AWS Lambda + SQS
- Интеграция с Anthropic Claude API (Sonnet 4.5)
- Поддержка Telegram Bot API

#### Lambda функции
- `webhook_lambda.py` - приём webhook от Telegram, отправка в SQS
- `worker_lambda.py` - обработка сообщений, генерация ответов
- `cleanup_function.py` - очистка старых данных (через DynamoDB TTL)

#### Утилиты
- `claude_utils.py` - интеграция с Claude API, базовая саммаризация
- `dynamo_utils.py` - CRUD операции для DynamoDB
- `telegram_utils.py` - отправка сообщений в Telegram

#### База данных (DynamoDB)
- Таблица **Users** - информация о пользователях
- Таблица **Channels** - группы и каналы
- Таблица **Threads** - треды в форумах
- Таблица **Messages** - история сообщений с TTL 1 год
- Таблица **Summaries** - сводки диалогов
- Таблица **Settings** - настройки режима работы

#### Режимы работы
- `/mode always` - всегда отвечать в группе
- `/mode mention` - только при упоминании
- `/mode off` - не отвечать

#### Scope в группах
- `/scope initiator` - учитывать только инициатора
- `/scope thread` - все участники треда
- `/scope hybrid` - приоритет инициатору, контекст других

#### Особенности
- Поддержка FIFO очередей с дедупликацией
- Умное обрезание контекста с сохранением приоритетов
- Базовая саммаризация диалогов
- Поддержка тредов, каналов, форумов
- Разделение сообщений на части до 4000 символов

#### Lambda Layer
- Docker-based сборка для независимости от ОС
- `Dockerfile` для Python 3.11
- `build_layer.bat` для Windows
- `build_layer.sh` для Linux/Mac
- `requirements.txt` с зависимостями

#### Документация
- `README.md` - общее описание проекта
- `README_ASYNC.md` - асинхронная архитектура
- `README_LAYER.md` - инструкции по Lambda Layer

### Технические детали
- Python 3.11
- AWS Lambda runtime
- Amazon SQS для асинхронной обработки
- Amazon DynamoDB для хранения данных
- Anthropic Claude API (Sonnet 4.5)
